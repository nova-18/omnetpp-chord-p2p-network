# CHORD-Based P2P Distributed Computing Network — OMNeT++

A discrete-event simulation of a **fully peer-to-peer distributed computing network** built with OMNeT++ 6.1. The system implements a **CHORD Distributed Hash Table (DHT)**-inspired routing protocol to achieve **O(log N) message complexity** for task routing across a ring topology, replacing the naive O(N) ring traversal with a finger-table-based approach.

---

## Overview

This is a fully peer-to-peer extension of the client-server simulation (Assignment 2) — server nodes are eliminated entirely. Each client node can both **assign** and **execute** subtasks. Tasks are divided into `x` subtasks (where `x > N`), and each subtask with ID `i` is routed to the client with ID `i % N`. Routing uses a CHORD finger table to reach the correct node in **O(log N)** hops instead of O(N). After task completion, results are propagated via a **gossip protocol** using SHA-256 message deduplication.

### Key Features
- **Pure P2P architecture** — no server nodes; all clients are peers
- **CHORD DHT-inspired routing** — finger table built per node, achieving O(log N) routing complexity
- **SHA-256 gossip deduplication** — each gossip message is hashed; duplicates are silently dropped
- **Dynamic subtask dispatch** — `x > N` subtasks distributed by `subtaskID % N` assignment rule
- **Thread-safe logging** — mutex-protected shared log file with simulation timestamps
- **Configurable topology** via `topology.txt` — no recompilation needed
- **Scheduled task start** — nodes stagger task creation to avoid initialization race conditions

## Configuration

### `topology.txt`
Edit this file to configure the network. **Do not modify any source code.**

```
parameters
{
clients = 7
}

client -> client delay
{
[100,100,100,100,100,100,100]
}
```

- `clients` — number of peer nodes in the ring
- Delay row defines per-link delays in **milliseconds** for each client's outgoing connections
- The number of delay values must match the number of clients declared

### `config.h`
Controls the number of task rounds per client:
```cpp
const int Total_Task_Count = 1;
```

## How to Run

1. **Install OMNeT++ 6.1** and ensure it is on your system PATH.

2. **Install dependencies:**
   ```bash
   sudo apt install libssl-dev python3   # Ubuntu/Debian
   ```

3. **Edit topology** (optional):
   ```bash
   nano topology.txt
   ```

4. **Generate the NED file** using the network manager script:
   ```bash
   python3 networkManager.py
   ```
   When prompted, enter the number of tasks to schedule (integer).

5. **Open the project in OMNeT++** and click **Run**, or build and run from terminal:
   ```bash
   make
   ./networksAssn3 -u Cmdenv omnetpp.ini
   ```

6. **Check output logs** after simulation:
   ```bash
   cat logs.log
   ```

> ⚠️ Always run `networkManager.py` **before** re-running the simulation in OMNeT++ to ensure the NED file is in sync with your topology.

## How It Works

### CHORD Finger Table
Each node `i` builds a finger table of size `log2(N)` at initialization:

```
finger[k] = (i + 2^k) % N,  for k = 0, 1, ..., log2(N)-1
```

When routing a subtask to destination `d`, the node finds the largest finger entry `≤ d` and forwards the message to that node — reducing worst-case hops from **O(N)** to **O(log N)**.

### Task Execution Flow
1. Each client generates a random integer array and divides it into `x > N` subtasks.
2. Subtask `i` is assigned to client `i % N` via CHORD routing.
3. The receiving node computes the local maximum and routes the response back to the originator via CHORD.
4. Once all subtask responses are received, the originator computes the global maximum and calls `sendGossip()`.

### Gossip Protocol
1. After task completion, each client generates a gossip message:
   ```
   <simTime>:<nodeName>:<taskNum>:<result>
   ```
2. The message is sent to all CHORD finger table neighbours.
3. On receipt, each node computes `SHA-256(gossipMessage)`. If unseen, it is forwarded to all neighbours except the sender and recorded in `messageList`.
4. Duplicate gossip messages (same hash) are silently dropped — preventing loops.
5. Once a node has received gossip from all `N` clients, it triggers the next task round (if `taskSent < Total_Task_Count`).

---

## Message Types

| Message | Description |
|---|---|
| `subtask` | Carries subarray data, source/dest IDs, routed via CHORD finger table |
| `subtaskResp` | Carries computed local maximum, routed back to originator via CHORD |
| `gossip` | Carries task result and gossip string, deduplicated via SHA-256 hash |
